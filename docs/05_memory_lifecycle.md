# 05 · 记忆生命周期（长期记忆的完整故事）

> 先读 `02_memory_schema.md`。本文件讲：记忆怎么进来、怎么沉淀、怎么被用、怎么过期。
> 这是 Demo "长期记忆"能力的核心，区别于"只存聊天记录"。

---

## 1. 全景：记忆的一生

```
                  ┌─────────────────────────────────────────┐
                  │            长期记忆 (LightMem + 偏好表)     │
                  └─────────────────────────────────────────┘
                       ▲           ▲              ▲        │
            ① 即时陈述  │  ② 离线总结 │  ③ 澄清学习    │        │ 检索
                       │           │              │        ▼
   用户明说偏好 ────────┘   当天对话 ──┘   一次ASK后 ───┘   决策层使用
   "我感冒了调高"       事后批量总结      把情景+选择沉淀    (能确定就用，
                      "爱安静"          "天热+恢复→25"      不能就再问)
                                                          │
                                                          ▼
                                             用过/情况变了 → 标记 expired
```

三条写入路径（①②③）+ 检索使用 + 过期，构成记忆的完整生命周期。Demo 要**全部体现**。

---

## 2. 路径① 即时陈述（在线，已有）

**时机**：用户在对话中**明说**一个偏好。
**做法**：当场解析，存一条 `source: user_stated` 的结构化偏好 + 调 LightMem `add_memory` 存对话原文。
**例**：用户"我感冒了，空调调高点" → 存 `{ac:26.5, condition:{health_state==sick}, source:user_stated}`。

这是 Demo 已有的能力，无需新增。

---

## 3. 路径② 离线总结（★ 本次新增重点）

### 是什么
不在对话当下抽取，而是在**当天对话结束后**，回看积累的整段对话，用 LLM **归纳**出用户没有明说、但行为透露的偏好。

这是**长期记忆系统的灵魂**：把零散对话"消化"成结构化认知（像人睡一觉把今天的经历整理成长期印象）。也是 LightMem 的核心能力——作者现成脚本里的 `topic_segment` / `text_summary` / `metadata_generate` / `update:"offline"` 配置就是干这个的。

### 用什么（复用 LightMem，不造轮子）
LightMem 的离线抽取。配置参考现成脚本的 `aggressive` 模式：
```python
{
  "topic_segment": True,      # 长对话按主题分段
  "text_summary": True,       # 段落总结
  "metadata_generate": True,  # 抽取结构化偏好
  "update": "offline",        # 离线更新
}
```
触发离线总结用：`add_memory(messages=当天对话, force_segment=True, force_extract=True)`。

### 流程
```
白天：用户每轮对话 → add_memory(...) 积累进 LightMem（不强制抽取，只积累）
       例：用户多次说"关音乐""太吵了""安静点"
        ↓
当天结束 / session 结束：触发离线总结
        ↓
LightMem 回看今天对话 → segment + summary + 抽取
        ↓
得到归纳偏好，如"用户开车偏好安静，倾向不放音乐"
        ↓
存入结构化偏好表：{music:off, condition:default, source:offline_summary,
                   evidence:"今天3次主动关闭音乐"}
```

### Demo 要把这一步"演出来"（关键）
离线总结不能悄悄发生，要让用户**看得见这个"消化"动作**。Demo 用**手动触发**（演示更可控）：
- 提供一个命令，如 `/summarize`（或 `/endday`）。
- 触发后显式打印：
```
💤 正在离线总结今天的对话……
📝 从今天的对话中总结出 1 条新偏好：
   「用户开车时偏好安静，倾向不主动播放音乐」
   依据：今天 3 次主动要求关闭音乐
   来源：离线总结  |  已存入长期记忆
```
> 真实产品里离线总结会自动定时跑；Demo 用手动命令，是为了能清楚地展示"离线总结"这一步的发生。

## 4. 路径③ 澄清学习（本次新增）

### 是什么
当一次 ASK（询问澄清）得到用户回答后，不只是执行，还把"**当时的情景 + 用户的选择**"沉淀成一条新偏好。这样**下次同情景，系统记得上次的选择，不用再问**。

这让"询问"从每次重复的成本，变成**一次性的学习投入**——问一次，以后就记住。

### 流程
```
5.5 天热，系统不确定 → ASK"感冒好了吗？好了25，没好26.5"
        ↓
用户答"好了" → 系统执行 set_ac(25)
        ↓
★ 沉淀新偏好：把"当时情景(天热+感冒恢复) + 选择(25)"存成一条
   {ac:25, condition:{health_state==recovering}, source:learned_from_clarification,
    evidence:"5.5天热且感冒恢复时，询问后用户选择25度"}
        ↓
6.10 又一次感冒后天热 → 检索召回这条 learned 偏好
        ↓
决策层判定：condition(感冒恢复)在当前情景明确成立、无冲突 → 可辨识 → 直接 ACT(25)，不再问
```

### condition 记多细（重要设计选择）
情景几乎不会完全重复（这次"天热+感冒恢复"，下次可能多了"晚上+有乘客"）。learned 偏好的 condition 若记太细则永远匹配不上、若太宽则过度泛化。

**Demo 阶段的规则：只记这次选择的关键决定因素（一两个主条件）。**
- 这里关键因素是"感冒状态=恢复" → condition 就记 `{health_state==recovering}`，忽略"天热""下午"等次要维度。
- 实现上：沉淀偏好时，可以让 LLM 判断"用户这次的选择主要由哪个条件决定"，取那个做 condition。简单起见，demo 也可由询问时的"那个区分维度"直接确定（系统问的就是"感冒好没好"，所以关键维度就是健康状态）。

## 5. 检索使用（决策层如何用上三种记忆）

不管偏好来自哪条路径，决策层（`03_decision_policy.md`）的使用方式**统一**：
1. 用户当前说话 → `lightmem.retrieve(query)` + 偏好表，召回相关的 **active** 偏好（三种 source 混在一起）。
2. 决策层判断可辨识性：
   - 召回的偏好里有一条 condition 在当前情景明确成立、无冲突 → ACT（**这条可能正是上次澄清学到的，于是"不用再问"**）。
   - 多条冲突/无法确定 → ASK。
3. **决策逻辑不区分 source**——learned 偏好和 user_stated 偏好一视同仁参与判断。这就是为什么"澄清学习"能自动让下次少问：它只是往记忆里多加了一条能匹配的偏好。

> 一个细节：`offline_summary` / `learned` 来的偏好可信度略低。Demo 可简单处理（一视同仁）；若想更讲究，可在决策时对低可信来源略保守（但 demo 不必，别增复杂度）。

---

## 6. 过期（记忆更新）

- 路径①的条件偏好，在条件解除时标 `expired`（感冒好了 → 感冒偏好过期）。
- 用户通过记忆面板手动删除/修正任何偏好（呼应"记忆可改"）。
- expired 的偏好不参与检索/决策。

## 7. 把三条路径串成一个完整 Demo 故事（多天）

```
[5.1] 用户"空调25度舒服"
      → 路径① 存 {ac:25, default, user_stated}

[5.3] 用户"我感冒了，调高点"
      → 路径① 存 {ac:26.5, 感冒时, user_stated}
      白天还几次说"关音乐""太吵"

[5.3 晚] /summarize 触发离线总结
      → 路径② 存 {music:off, default, offline_summary, 依据"3次关音乐"}

[5.5] 天热上车"好热"
      → 检索到 {25,默认} 和 {26.5,感冒时} 冲突 → ASK"感冒好没"
      → 用户"好了" → ACT set_ac(25) + 把感冒偏好标 expired
      → 路径③ 学到 {ac:25, 感冒恢复, learned_from_clarification}

[5.7] 上车
      → 检索到 {music:off,offline_summary} → 不主动放音乐（应用了离线总结的偏好）

[6.10] 又一次感冒后天热上车
      → 检索到 {ac:25, 感冒恢复, learned} → 直接 ACT set_ac(25)，不再问
        （应用了上次澄清学到的偏好）
```

这条多天故事，把**三条记忆路径 + 检索使用 + 过期**全演到了。这才是一个完整的长期记忆系统。

---

## 8. 验收（在 03/04 的验收基础上新增）

- **离线总结可见**：`/summarize` 后能看到从对话归纳出的新偏好（source=offline_summary），且它确实是 LLM 总结的、不是硬编。
- **澄清学习生效**：同一情景出现两次，第一次 ASK、**第二次直接 ACT 不问**（因为第一次学到了）。
- **多来源共存**：记忆面板里能看到三种 source 的偏好并存，且都能正确参与决策。
