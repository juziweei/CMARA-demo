# 09 · 给 Codex / 接手人继续深挖项目的提示词模板

> 这份文档不是项目说明，而是“怎么问”。
> 如果后续有人要继续用 Codex 深挖这个项目，下面这些模板比泛泛地说“帮我看看项目”有效得多。

---

## 1. 使用原则

如果目标是让 Codex 深入研究这个项目，请尽量在提示词里明确：

1. **以服务器为准**
2. **只读还是允许改代码**
3. **你关心的是工程、运行、前端接手、还是算法质量**
4. **要输出什么文档或结论**

最好避免这种模糊说法：

```text
帮我看看这个项目
```

因为它太宽，容易得到泛泛总结。

更有效的是：

```text
以服务器上的 /root/vehicle_memory_demo 为准，只读分析。
目标是让我能把项目移交给接手做前端 demo 的同事。
请重点梳理运行环境、后端主链路、数据路径、可复现命令、当前功能边界和已知问题。
最后输出一份 server handoff 文档大纲。
```

---

## 2. 模板：项目级只读梳理

```text
以服务器上的 /root/vehicle_memory_demo 为准，只读分析整个项目，不要改代码。

目标：
让我完整知道这个项目现在做到了什么，以及接手人如何快速继续推进。

请重点梳理：
1. 服务器环境和依赖位置
2. 配置文件和默认值
3. 后端主链路（输入、记忆、检索、决策、学习、离线抽取）
4. 数据落盘路径
5. 可复现命令
6. 当前功能边界、已知问题、非目标

最后输出：
- 一份项目总览
- 一份 server runbook
- 一份接手建议
```

适用场景：

- 新同事接手前
- 你自己过几天回来继续做之前

---

## 3. 模板：后端闭环深挖

```text
以服务器上的 /root/vehicle_memory_demo 为准，只读分析这个 demo 的后端闭环。

请围绕这些模块梳理：
- DemoSession
- LightMemStore
- PreferenceTable
- Policy
- ClarificationLearner
- OfflineSummarizer
- DirectPreferenceExtractor

要求：
1. 逐步说明用户输入后数据怎么流动
2. 说明每一步状态在哪里变化
3. 说明哪些数据会落盘
4. 说明下一次相似场景为什么能复用偏好
5. 判断哪些环节已经闭环，哪些仍有缺口

输出：
- 一份后端闭环说明
- 一份验收清单
```

适用场景：

- 你想确认后端是否真的收口
- 你想让 Codex 先分析再决定要不要继续改

---

## 4. 模板：前端接手视角

```text
以服务器上的 /root/vehicle_memory_demo 为准，只读分析，不改代码。

站在接手做前端 demo 的工程师视角，告诉我：
1. 当前后端已经完成了什么
2. 哪个入口最适合前端接
3. 后端能返回哪些核心状态
4. 页面最该展示什么
5. 哪些东西先不要做

最后输出一份 frontend handoff 文档，要求只写服务器路径和服务器命令，不要混本地路径。
```

适用场景：

- 你要把项目交给前端同事
- 你希望前端少踩后端边界上的坑

---

## 5. 模板：服务器运行手册

```text
只读分析服务器环境，生成一份可复现运行手册。

必须包含：
1. ssh 登录方式
2. 项目根目录
3. Python 环境路径
4. PYTHONPATH 设置方式
5. 核心配置文件位置
6. 数据目录位置
7. scenario / benchmark 的运行命令
8. reset 方法
9. 常见排障方法

要求：
- 只写服务器侧信息
- 不要写本地开发路径
```

适用场景：

- 需要别人直接上服务器跑 demo
- 需要交付运维/测试/前端协同同事

---

## 6. 模板：benchmark / 偏好质量分析

```text
只读分析 benchmark 脚本和服务器结果，不要改代码或 ground truth。

目标：
弄清楚当前偏好抽取质量问题在哪里。

请重点分析：
1. expected 和 actual 的逐条差异
2. 哪些 condition 是用户明确说出来的
3. 哪些 condition 是模型脑补的
4. 哪些冲突来自 schema 太窄
5. 哪些冲突来自模型误抽

最后输出：
- 一张 expected vs actual 对照表
- 一份 condition 标注裁决建议
- 一份冲突分析
```

适用场景：

- benchmark 结果卡在 80%-90%
- 你不确定该改代码还是该改标注

---

## 7. 模板：允许改代码的收口任务

```text
以服务器上的 /root/vehicle_memory_demo 为准，允许改代码。

目标：
收口 demo 后端闭环，优先保证工程稳定和可演示，不追求完美抽取。

约束：
- 不动 LightMem add_memory / retrieve
- 不改 benchmark ground truth
- 不扩产品功能

验收标准：
1. 用户输入会写入 LightMem
2. 缺关键维度时会 ask_user
3. 用户回答后会新增或更新 PreferenceTable
4. 当天长对话可离线抽取偏好入表
5. 下一次相似请求能利用已学偏好，不重复追问
6. 本地 pytest 通过，服务器 scenario 跑通

最后输出：
- 代码改动
- 验证结果
- 服务器复现命令
- 已知问题
```

适用场景：

- 要继续做工程收口
- 不想让 Codex跑偏去做大重构

---

## 8. 最推荐的写法

如果你只有一句话的机会，推荐写成：

```text
以服务器上的 /root/vehicle_memory_demo 为准，先只读分析。
我的目标是把这个项目完整移交给接手做前端 demo 的同事。
请重点梳理：运行环境、后端主链路、数据路径、可复现命令、当前功能边界、已知问题。
要求只写服务器侧信息，不要混本地路径。
```

这类提示词的好处是：

- 目标明确
- 范围明确
- 环境明确
- 输出物明确

更容易得到高质量文档，而不是泛泛总结。
