# 11 · 下一阶段 Goal

## Goal

完成 2026-07-02 前端展示与对外沟通准备：把车载长期记忆 demo 包装成一套能同时面向技术评审、论文方向讨论、投资人/资源方沟通的材料，并在展示后进入可复现、可评测、可试点的下一阶段。

## Scope

本阶段优先服务四件事：

1. 前端能稳定展示长期记忆闭环：
   - `turn -> retrieve -> ask/act -> clarification -> learn -> summarize -> reuse`
2. 文档能讲清技术路线：
   - LightMem 检索
   - 结构化偏好表
   - 条件偏好激活
   - ACT / ASK 决策
   - 澄清学习
3. 文档能支撑论文/技术报告方向：
   - 任务定义
   - 研究问题
   - 评测指标
   - baseline / ablation 方向
4. 文档能支撑资源沟通：
   - 云服务和 GPU 推理
   - 数据集与标注
   - 前后端工程
   - 车载场景和试点合作

核心材料：

- `docs/13_showcase_technical_route_and_resource_plan.md`

## 0. 本阶段的真实目标

当前展示不再只追求“能跑”，而是要同时满足三类受众：

1. 技术评审
   - 看懂系统如何记忆、检索、追问、学习、复用。
2. 论文讨论
   - 看懂问题定义、研究路线、可评测对象，以及为什么这不是普通 RAG。
3. 投资人 / 资源方
   - 看懂产品价值、数据壁垒、算力需求、试点路径，以及为什么值得继续投入。

因此前端必须从“调试控制台”升级成“可讲故事的演示驾驶舱”，重点不是按钮数量，而是信息组织方式。

## 1. 分阶段推进路线

### Stage A: 展示叙事定稿

先把一页 demo 的叙事结构固定下来，避免前端先画 UI 以后又反复推倒。

必须明确：

1. 三套讲法
   - 技术版
   - 论文版
   - 投资人版
2. 一个主叙事
   - 长期记忆不是“记住一句话”，而是“知道哪条记忆该在什么时候被激活”。
3. 一个演示闭环
   - `turn -> retrieve -> ask/act -> clarification -> learn -> summarize -> reuse`

### Stage B: 英文长场景数据集

前端不能只靠一个 family trip 场景。需要一个可以轮播的英文长场景库，至少覆盖：

1. Family Coastal Trip
2. Morning Commute Under Time Pressure
3. Elderly Passenger Comfort
4. EV Range Anxiety Trip
5. Work Call Privacy Mode

每个场景都要有：

- 长英文输入
- 多轮问答
- 至少一次 ASK
- 至少一次 clarification learning
- 至少一次 summarize
- 至少一次 future reuse

### Stage C: 前端信息架构重做

前端要从“左聊天右 trace”的调试布局，改成四层展示：

1. Scenario Gallery
   - 场景选择、演示进度、数据集轮播。
2. Conversation Theater
   - 英文长对话，支持自动播放和手动输入。
3. Memory Timeline
   - 展示记忆来自哪里、何时写入、何时过期。
4. Decision Flow
   - 展示系统为什么 ASK / 为什么 ACT / 为什么学习。

### Stage D: 文档与讲解词同步

前端重做前，文档先统一口径：

- 30 秒版本
- 2 分钟版本
- 5 分钟版本
- PPT 页纲
- 风险边界
- 资源请求

## 2. 交付拆分

### 2.1 第一批交付

1. 新增英文场景草案文档。
2. 新增前端改版草图说明。
3. 新增演示用数据规范。

### 2.2 第二批交付

1. 前端加入场景切换。
2. 前端加入更强的状态展示。
3. 前端加入可视化的 memory / decision 流程。

### 2.3 第三批交付

1. 场景覆盖扩展到 10 个。
2. 自动播放与手动编辑共存。
3. 稳定录屏和现场演示流程。

## 3. English Data Rules

为了让对外展示更统一，后续新增的场景与问答必须全部使用英文。

要求：

1. 问题要长一点，像真实用户在车内说话，不要只写短句。
2. 回答要完整，能体现澄清、确认、纠错、复用。
3. 尽量包含上下文信息，例如家庭成员、时间压力、健康状态、路况、隐私需求。
4. 每个场景至少包含一条冲突偏好和一条可学习偏好。
5. 用词要自然，不要写成论文摘要。

## 4. 前端演示重点

当前前端不要继续只强调：

- API 是否在线
- trace JSON 是否展开

而要强调：

- 当前是什么场景
- 当前系统记住了什么
- 当前系统为什么不确定
- 当前系统问了哪个关键维度
- 当前系统学到了什么
- 下一次为什么少问

## 5. 验收标准补充

除了原有验收标准，本阶段还要满足：

1. 至少有 5 个英文场景可切换。
2. 每个场景都有长英文对话脚本。
3. 每个场景都能展示 memory -> decision -> learning 的闭环。
4. 前端布局能同时容纳 story、memory、decision、trace 四类信息。
5. 对外讲解词能覆盖技术、论文、资源三种受众。

## Acceptance

展示前：

1. 前端页面可打开并连接 API。
2. `一键全流程` 能完整跑通。
3. 偏好表能展示 `user_stated`、`offline_summary`、`learned_from_clarification` 等来源。
4. Decision Trace 能展示当前轮为什么 ASK / 为什么 ACT。
5. README 能清楚指向展示材料。
6. 展示材料包含：
   - 现场流程
   - 讲解词
   - 技术路线
   - 论文角度
   - 资源需求
   - PPT 结构
   - 风险边界

展示后 1-2 周：

1. 固化服务器部署方式，减少临场启动风险。
2. 增加 10 个稳定车载场景脚本。
3. 开始整理条件偏好评测集。
4. 明确云服务、数据标注和车载接口合作需求。
5. 准备一版技术报告或论文初稿大纲。

## Non-goals

- 不在当前 demo 仓库里做完整论文实验管线。
- 不把 mock 车控直接包装成量产能力。
- 不承诺当前系统已经支持多用户、多车辆、多租户。
- 不为了展示而硬编码新的假场景；展示必须基于真实 API 和现有闭环。
