## 1. 一句话定位

这是一个面向车载智能助手的长期记忆系统 demo。它展示的不是普通聊天，也不是单次车控，而是：

```text
用户长期交互
-> 系统沉淀条件化偏好
-> 当前场景检索相关记忆
-> 能确定就执行，不能确定就追问
-> 追问后把选择学成新偏好
-> 下次相似场景减少重复打扰
```

更适合对外讲成：

> 我们在做的不是一个"会聊天的车机"，而是一个能长期理解驾驶者和家庭出行习惯的车载记忆决策层。它让车从被动执行命令，变成能在不确定时谨慎确认、在确认后持续学习的个性化智能体。

## 2. 明天展示的主线

### 2.1 展示目标

明天前端展示要让观众看到四件事：

1. 系统真的有长期记忆，不只是当前 prompt 里的聊天记录。
2. 系统不是盲目自动化，遇到条件冲突会主动追问。
3. 追问不是一次性问答，回答会变成下次可复用的偏好。
4. 前端能解释系统为什么 ASK / 为什么 ACT，降低黑盒感。

### 2.2 推荐现场流程

准备：

```bash
python3 scripts/run_api_server.py --host 0.0.0.0 --port 8010
cd frontend
python3 -m http.server 8080
```

浏览器打开：

```text
http://127.0.0.1:8080
```

演示顺序：

1. 点击 `检查连接`
   - 证明前端连接真实后端，不是静态页面。

2. 点击 `一键全流程`
   - 页面会自动播种家庭出游场景、触发离线总结、产生追问、学习澄清结果、复现下一次直接执行。

3. 重点讲右侧 `Decision Trace`
   - 展示候选偏好、召回偏好、决策 JSON、工具结果。
   - 讲清楚系统不是"猜"，而是在做可解释的可辨识性判断。

4. 重点讲左侧 `偏好表`
   - 展示 `source` 字段：`user_stated`、`offline_summary`、`learned_from_clarification`。
   - 这三个来源是长期记忆系统的关键，不是普通聊天记录。

5. 手动输入一个相似场景
   - 例如：`下周一家人又要去郊游了，感冒恢复了，还是有点热。`
   - 观察系统是否更倾向直接执行，体现"上次问过，这次少问"。

### 2.3 现场讲解词

30 秒版本：

```text
这个 demo 展示的是车载长期记忆闭环。用户不是每次上车都从零开始，系统会记住偏好、检索相关历史，并判断当前信息是否足够。如果足够，它直接执行；如果多个偏好冲突，它不会瞎猜，而是追问关键条件。更重要的是，追问后的回答会沉淀成新的偏好，下次类似场景就能减少重复打扰。
```

2 分钟版本：

```text
现在很多车载助手的问题是，它能执行命令，但不真正理解用户长期习惯。比如用户平时喜欢 25 度，但感冒时更喜欢 26.5 度。过几天用户说"好热啊"，系统如果直接设 25 度可能忽略健康状态，直接设 26.5 度也可能过度保守。

我们的核心不是让模型更会聊天，而是加一个长期记忆和决策层：先从历史交互里检索相关偏好，再看这些偏好的触发条件在当前场景里能不能判断。如果不能判断，就问"您感冒好些了吗？好了我设 25 度，还没好我设 26.5 度"。用户回答后，系统执行动作，并把这次澄清结果写回长期记忆。

前端右侧的 Decision Trace 会显示系统召回了什么、为什么需要追问、执行了什么工具、学到了什么。这个可解释过程是未来车载智能助手进入真实产品的重要前提，因为车控不是纯聊天，错误自动化会损害用户信任。
```

5 分钟版本结构：

1. 痛点：车载助手缺少长期个性化，单次对话很强，跨天记忆很弱。
2. 技术路线：LightMem 检索 + 结构化偏好表 + ACT/ASK 决策 + 澄清学习。
3. Demo：家庭出游、天气热、健康状态冲突、追问、学习、复用。
4. 论文角度：这是条件偏好激活问题，不是简单 RAG。
5. 资源需求：算力、数据、评测、前端工程、车载集成合作。

## 3. 项目背景与问题定义

### 3.1 背景

车载 AI 的价值不只在"能听懂一句话"，而在长期使用中逐渐适应用户：

- 不同驾驶者对温度、音乐、座椅、导航风格的偏好不同。
- 同一个驾驶者在不同状态下偏好也不同，例如感冒、带孩子、长途、夜间、赶时间。
- 家庭用车场景里，偏好经常受到乘客、健康状态、出行目的影响。

因此，车载助手需要处理的是条件化、动态、可更新的偏好，而不是静态用户画像。

### 3.2 当前行业常见问题

普通 LLM 车载助手容易卡在三类问题：

1. **记忆短**：跨天、多 session 的偏好难以稳定复用。
2. **记忆粗**：只记"用户喜欢 25 度"，不记"什么条件下喜欢 25 度"。
3. **自动化冒进**：有冲突偏好时直接猜，导致用户觉得系统不可靠。

这个项目的技术重点是第三点和第二点的结合：

```text
长期记忆不是最终目标。
能够在当前场景里正确激活哪条记忆，才是车载长期记忆真正有价值的地方。
```

## 4. 技术路线

### 4.1 当前系统分层

```text
前端展示层
  - 聊天区
  - 偏好表
  - 状态摘要
  - Decision Trace

HTTP API / DemoSession
  - /turn
  - /clarification
  - /summarize
  - /preferences
  - /demo/family_trip

决策层 Policy
  - 输入当前上下文 + 召回偏好
  - 输出 ACT / ASK / REPLY
  - 关键：判断当前情境是否足以激活某条条件偏好

记忆层
  - LightMem 存原始交互与 embedding 检索
  - PreferenceTable 存结构化偏好、condition、source、status

执行层
  - set_ac_temperature
  - ask_user
  - general_chat
```

### 4.2 三条记忆写入路径

| 路径 | 发生时机 | 示例 | 对外讲法 |
| --- | --- | --- | --- |
| 用户即时陈述 | 用户明说偏好时 | "我感冒了，空调调高点" | 用户明确告诉系统的记忆 |
| 离线总结 | 一段对话结束后 | 多次说"别放音乐" -> 总结出家庭出游偏安静 | 系统把零散行为消化成长期偏好 |
| 澄清学习 | 系统追问后 | 问"感冒好了吗" -> 用户答"好了" -> 学到恢复状态下 25 度 | 追问变成一次性学习成本 |

### 4.3 为什么不是普通 RAG

普通 RAG 更像：

```text
当前问题 -> 检索相关文本 -> 拼进 prompt -> 生成回答
```

本项目多了三个关键结构：

1. **结构化偏好表**
   - 记忆不是只有文本，还有 `preference`、`value`、`condition`、`status`、`source`。

2. **条件激活**
   - 系统要判断当前场景是否满足某条偏好的触发条件。

3. **不确定时的动作选择**
   - 不确定不是继续生成废话，而是选择 `ASK`，并提出最小必要追问。

因此对外可以说：

> RAG 解决"想起什么"，本项目进一步解决"什么时候该用、什么时候不该用、什么时候必须先问"。

### 4.4 为什么适合车载场景

车载场景和普通聊天不同：

- 动作有真实后果，例如温度、座椅、音乐、导航路线。
- 用户在驾驶中不适合被频繁打扰。
- 错误自动化会快速破坏信任。
- 家庭出行有多人偏好冲突，需要解释和确认。

所以车载助手不能只追求"更主动"，还要追求"主动得有边界"：

```text
低风险、条件明确 -> 自动执行
偏好冲突、条件缺失 -> 主动追问
追问后 -> 学习，减少下次打扰
```

## 5. 论文/研究角度

### 5.1 可包装的研究问题

本项目可以从论文角度表述为：

> 在长期多轮车载交互中，助手如何从历史对话中抽取条件化用户偏好，并在新场景下选择性激活正确偏好；当条件不可判定时，如何通过最小澄清问题降低错误自动化风险，并把澄清结果写回长期记忆。

更短的论文问题：

```text
Conditional Preference Activation for Long-Term Vehicle Memory Agents
```

中文：

```text
面向车载长期记忆智能体的条件偏好选择性激活
```

### 5.2 与已有研究的关系

可引用的研究脉络：

1. RAG 证明了外部检索能增强模型知识访问，但它主要解决"检索相关信息"问题。
   - Lewis et al., 2020, Retrieval-Augmented Generation.
   - https://arxiv.org/abs/2005.11401

2. MemoryBank、MemGPT 说明长期记忆和分层记忆对长程交互有价值。
   - MemoryBank: https://arxiv.org/abs/2305.10250
   - MemGPT: https://arxiv.org/abs/2310.08560

3. LoCoMo 表明长程对话记忆仍然困难，尤其是多 session、时间和事件一致性。
   - LoCoMo: https://arxiv.org/abs/2402.17753

4. LightMem 的方向和本项目底层一致：把在线交互与离线记忆整理分开，以降低长期记忆系统成本。
   - LightMem: https://arxiv.org/abs/2510.18866
   - Lightweight LLM Agent Memory with Small Language Models: https://arxiv.org/abs/2604.07798

5. Reflexion 支持一个重要思想：不一定要更新模型权重，也可以通过外部记忆和反馈改善后续行为。
   - Reflexion: https://arxiv.org/abs/2303.11366

本项目的差异点：

```text
已有工作多关注"记住和召回"。
本项目更关注"车载动作前，应该激活哪条条件偏好；不确定时是否应该追问"。
```

### 5.3 可形成论文贡献的方向

可以把后续研究贡献拆成三层：

1. **任务定义**
   - 定义车载长期记忆中的条件偏好激活任务。
   - 输出不只是回答文本，而是 `apply / abstain / clarify` 或 `ACT / ASK / REPLY`。

2. **数据集/评测**
   - 构造多天、多场景、多条件冲突的车载偏好数据。
   - 标注每一轮应该应用哪条偏好、应该追问哪个缺失条件、哪些旧偏好应过期。

3. **方法**
   - LightMem 召回历史证据。
   - 结构化偏好表显式表示 condition。
   - Policy 做选择性激活和追问决策。
   - Clarification Learner 把追问结果写回长期记忆。

### 5.4 可以评测的指标

论文/技术报告里可以考虑这些指标：

| 维度 | 指标 | 意义 |
| --- | --- | --- |
| 偏好抽取 | preference extraction F1 | 是否从对话中抽对偏好 |
| 条件识别 | condition accuracy | 是否识别偏好的触发条件 |
| 选择性激活 | apply / abstain / clarify accuracy | 是否该用时用，不该用时不乱用 |
| 追问质量 | missing dimension accuracy | 是否问到了真正缺失的关键条件 |
| 学习效果 | repeated clarification reduction | 同类场景第二次是否减少追问 |
| 安全性 | unsafe / wrong action rate | 是否在条件不明时错误执行 |
| 体验 | average user turns to action | 到达正确动作需要几轮 |

### 5.5 当前 demo 和论文工程的边界

当前仓库是 demo 工程，不是完整论文实验管线。

当前已经能证明：

- 闭环可跑通。
- 前端可解释。
- LightMem + PreferenceTable + Policy 的系统结构成立。
- ASK -> clarification -> learn -> reuse 的产品逻辑成立。

后续论文还需要：

- 更大规模数据。
- 更严格标注。
- 基线方法。
- 消融实验。
- 跨模型、跨场景评测。

## 6. 对投资人/资源方的叙事

### 6.1 价值主张

车载智能助手的竞争点会从"能不能回答"转向"能不能长期适应我"。这个项目切的是个性化车载 AI 的中间层：

```text
LLM 负责理解语言
车控系统负责执行动作
我们负责长期记忆、条件偏好和谨慎决策
```

这是一个有产品价值的中间层，因为它可以服务多个上层场景：

- 空调/座椅/香氛/音乐等舒适性控制。
- 家庭出行偏好。
- 通勤路线和导航风格。
- 车内多用户偏好协调。
- 驾驶者状态变化下的个性化辅助。

### 6.2 为什么现在做

1. 车企和座舱系统已经开始接入大模型，但大多数仍停留在单轮问答或功能调用。
2. 用户真正愿意长期使用的助手，需要记得住、解释得清、错了能改。
3. 车载场景天然有高频重复任务，长期记忆能明显降低交互成本。
4. 当前 demo 已经有可展示闭环，可以从 prototype 快速推进到 pilot。

### 6.3 护城河方向

这个项目未来的护城河不只是模型，而是四类资产：

1. **车载条件偏好数据**
   - 多天、多用户、多乘客、多场景。
   - 含冲突、过期、澄清、复用。

2. **评测体系**
   - 能衡量该问不问、该做不做、是否减少重复打扰。

3. **车载场景 schema**
   - 健康状态、乘客状态、天气、出行目的、驾驶状态、时间、地点等条件维度。

4. **工程闭环**
   - 前端可解释、后端可复现、API 可接入、记忆可更新。

## 7. 下一阶段需要的资源支持

### 7.1 算力资源

目标不是盲目堆 GPU，而是分层配置：

| 阶段 | 用途 | 建议资源 |
| --- | --- | --- |
| Demo 稳定展示 | 跑 Qwen 7B/14B、LightMem、API、前端 | 1 台带单卡 GPU 的云服务器或实验室服务器 |
| 数据生成/批量评测 | 生成多场景对话、跑抽取、跑基线 | 可弹性申请多卡或多实例批处理 |
| 小模型/策略模型探索 | 训练 preference extractor、reranker 或 policy classifier | 1-4 卡 GPU，按实验规模弹性扩展 |
| 产品试点 | 多用户 session、日志、监控、容灾 | 云上 API 服务 + 独立向量库/对象存储/数据库 |

当前 demo 优先级：

1. 保证一台稳定服务器能持续跑前端展示。
2. 把 LLM inference、API、前端服务拆成可重启、可监控的服务。
3. 再考虑批量评测和训练资源。

### 7.2 云服务资源

建议拆成这些基础服务，不要一开始做复杂平台：

| 资源 | 用途 | 当前优先级 |
| --- | --- | --- |
| GPU 推理实例 | 部署 Qwen / vLLM | 高 |
| CPU API 实例 | 跑 HTTP API、session runtime | 中 |
| 向量数据库 | 存 embedding 召回索引，当前可先用本地 qdrant | 中 |
| 对象存储 | 保存数据集、日志、实验结果、截图视频 | 中 |
| 关系数据库 | 存 preference table、用户、session、审计日志 | 中 |
| 监控日志 | 展示现场排障、后续 pilot 观测 | 中 |
| CI/CD | 测试、部署、回滚 | 低到中 |

明天展示不需要把这些都做完，但需要向资源方说清楚：

```text
当前 demo 是单 session、单服务器形态。
下一阶段要从可演示系统升级为可试点系统，需要云上推理、持久化存储、日志监控和数据闭环。
```

### 7.3 数据资源

这是后续最关键的资源。

需要三类数据：

1. **合成车载多天对话**
   - 低成本启动。
   - 用于覆盖极端场景、条件冲突、过期、澄清。

2. **人工标注评测集**
   - 每条样本标注：
     - 当前上下文
     - 历史偏好
     - 正确动作：ACT / ASK / REPLY
     - 正确工具参数
     - 缺失条件
     - 是否应过期某条偏好

3. **真实用户试点数据**
   - 来自封闭测试或模拟座舱。
   - 重点收集用户纠错、拒绝、手动修改记忆的行为。

数据规模建议：

| 阶段 | 数据目标 |
| --- | --- |
| Demo 后 2 周 | 50-100 条高质量多天场景，用于稳定前端和后端 |
| 技术报告/论文初版 | 500-1000 条标注样本，覆盖 5-8 类车载控制 |
| pilot 前 | 多用户、多车型、多地域的真实或半真实日志 |

### 7.4 人力资源

最小团队配置：

| 角色 | 工作 |
| --- | --- |
| 算法/LLM 工程 | Policy、偏好抽取、评测、prompt/模型优化 |
| 后端工程 | API、session、存储、部署、日志 |
| 前端/交互设计 | 展示台、记忆可解释、用户可编辑 |
| 数据标注/产品 | 场景设计、标签规范、质量审查 |
| 车载系统合作方 | 车控接口、座舱约束、真实场景验证 |

当前最缺：

1. 高质量场景数据和标注规范。
2. 稳定云端推理服务。
3. 能把 demo 变成长期可试用产品的工程资源。

  - full system
- 做消融：

  - 去掉 offline summary

  - 去掉 expired status

### 8.3 长期：3-6 个月

- 多用户、多车、多 session。
- 与真实或模拟车控 API 对接。

## 11. 风险与边界

必须主动说明边界，这会让项目更可信：

- 当前是 demo，不是量产级车载系统。
- 当前单 session，不是多用户平台。
- 当前车控函数是 mock，不直接控制真实车辆。
- 当前 condition schema 还窄，需要数据驱动扩展。
- 当前论文评测还没完整建立，不能把 demo 结果等同于论文结论。
- 当前 `music_mode` 可记忆和展示，但没有真实执行工具。

推荐说法：

```text
我们现在完成的是最关键的闭环验证：记忆、检索、判断、追问、学习、复用。下一阶段不是证明一个页面好看，而是把这个闭环扩展到更多车载场景和更严格评测。
```
